<div id="lxrcode"><pre>  <a name="L1" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L1">1</a> REDUCING OS JITTER DUE TO PER-CPU KTHREADS
  <a name="L2" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L2">2</a> 
  <a name="L3" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L3">3</a> This document lists per-CPU kthreads in the Linux kernel and presents
  <a name="L4" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L4">4</a> options to control their OS jitter.  Note that non-per-CPU kthreads are
  <a name="L5" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L5">5</a> not listed here.  To reduce OS jitter from non-per-CPU kthreads, bind
  <a name="L6" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L6">6</a> them to a "housekeeping" CPU dedicated to such work.

 <a name="L33" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L33">33</a> KTHREADS
 <a name="L34" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L34">34</a> 
 <a name="L35" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L35">35</a> Name: ehca_comp/%u
 <a name="L36" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L36">36</a> Purpose: Periodically process Infiniband-related work.
 <a name="L37" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L37">37</a> To reduce its OS jitter, do any of the following:
 <a name="L38" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L38">38</a> 1.      Don't use eHCA Infiniband hardware, instead choosing hardware
 <a name="L39" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L39">39</a>         that does not require per-CPU kthreads.  This will prevent these
 <a name="L40" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L40">40</a>         kthreads from being created in the first place.  (This will
 <a name="L41" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L41">41</a>         work for most people, as this hardware, though important, is
 <a name="L42" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L42">42</a>         relatively old and is produced in relatively low unit volumes.)
 <a name="L43" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L43">43</a> 2.      Do all eHCA-Infiniband-related work on other CPUs, including
 <a name="L44" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L44">44</a>         interrupts.
 <a name="L45" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L45">45</a> 3.      Rework the eHCA driver so that its per-CPU kthreads are
 <a name="L46" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L46">46</a>         provisioned only on selected CPUs.
 <a name="L47" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L47">47</a> 
 <a name="L48" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L48">48</a> 
 <a name="L49" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L49">49</a> Name: irq/%d-%s
 <a name="L50" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L50">50</a> Purpose: Handle threaded interrupts.
 <a name="L51" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L51">51</a> To reduce its OS jitter, do the following:
 <a name="L52" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L52">52</a> 1.      Use irq affinity to force the irq threads to execute on
 <a name="L53" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L53">53</a>         some other CPU.
 <a name="L54" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L54">54</a> 
 <a name="L55" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L55">55</a> Name: kcmtpd_ctr_%d
 <a name="L56" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L56">56</a> Purpose: Handle Bluetooth work.
 <a name="L57" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L57">57</a> To reduce its OS jitter, do one of the following:
 <a name="L58" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L58">58</a> 1.      Don't use Bluetooth, in which case these kthreads won't be
 <a name="L59" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L59">59</a>         created in the first place.
 <a name="L60" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L60">60</a> 2.      Use irq affinity to force Bluetooth-related interrupts to
 <a name="L61" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L61">61</a>         occur on some other CPU and furthermore initiate all
 <a name="L62" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L62">62</a>         Bluetooth activity on some other CPU.
 <a name="L63" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L63">63</a> 
 <a name="L64" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L64">64</a> Name: ksoftirqd/%u
 <a name="L65" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L65">65</a> Purpose: Execute softirq handlers when threaded or when under heavy load.
 <a name="L66" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L66">66</a> To reduce its OS jitter, each softirq vector must be handled
 <a name="L67" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L67">67</a> separately as follows:
 <a name="L68" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L68">68</a> TIMER_SOFTIRQ:  Do all of the following:
 <a name="L69" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L69">69</a> 1.      To the extent possible, keep the CPU out of the kernel when it
 <a name="L70" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L70">70</a>         is non-idle, for example, by avoiding system calls and by forcing
 <a name="L71" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L71">71</a>         both kernel threads and interrupts to execute elsewhere.
 <a name="L72" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L72">72</a> 2.      Build with CONFIG_HOTPLUG_CPU=y.  After boot completes, force
 <a name="L73" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L73">73</a>         the CPU offline, then bring it back online.  This forces
 <a name="L74" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L74">74</a>         recurring timers to migrate elsewhere.  If you are concerned
 <a name="L75" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L75">75</a>         with multiple CPUs, force them all offline before bringing the
 <a name="L76" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L76">76</a>         first one back online.  Once you have onlined the CPUs in question,
 <a name="L77" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L77">77</a>         do not offline any other CPUs, because doing so could force the
 <a name="L78" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L78">78</a>         timer back onto one of the CPUs in question.
 <a name="L79" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L79">79</a> NET_TX_SOFTIRQ and NET_RX_SOFTIRQ:  Do all of the following:
 <a name="L80" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L80">80</a> 1.      Force networking interrupts onto other CPUs.
 <a name="L81" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L81">81</a> 2.      Initiate any network I/O on other CPUs.
 <a name="L82" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L82">82</a> 3.      Once your application has started, prevent CPU-hotplug operations
 <a name="L83" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L83">83</a>         from being initiated from tasks that might run on the CPU to
 <a name="L84" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L84">84</a>         be de-jittered.  (It is OK to force this CPU offline and then
 <a name="L85" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L85">85</a>         bring it back online before you start your application.)
 <a name="L86" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L86">86</a> BLOCK_SOFTIRQ:  Do all of the following:
 <a name="L87" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L87">87</a> 1.      Force block-device interrupts onto some other CPU.
 <a name="L88" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L88">88</a> 2.      Initiate any block I/O on other CPUs.
 <a name="L89" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L89">89</a> 3.      Once your application has started, prevent CPU-hotplug operations
 <a name="L90" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L90">90</a>         from being initiated from tasks that might run on the CPU to
 <a name="L91" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L91">91</a>         be de-jittered.  (It is OK to force this CPU offline and then
 <a name="L92" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L92">92</a>         bring it back online before you start your application.)
 <a name="L93" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L93">93</a> BLOCK_IOPOLL_SOFTIRQ:  Do all of the following:
 <a name="L94" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L94">94</a> 1.      Force block-device interrupts onto some other CPU.
 <a name="L95" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L95">95</a> 2.      Initiate any block I/O and block-I/O polling on other CPUs.
 <a name="L96" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L96">96</a> 3.      Once your application has started, prevent CPU-hotplug operations
 <a name="L97" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L97">97</a>         from being initiated from tasks that might run on the CPU to
 <a name="L98" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L98">98</a>         be de-jittered.  (It is OK to force this CPU offline and then
 <a name="L99" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L99">99</a>         bring it back online before you start your application.)
<a name="L100" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L100">100</a> TASKLET_SOFTIRQ: Do one or more of the following:
<a name="L101" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L101">101</a> 1.      Avoid use of drivers that use tasklets.  (Such drivers will contain
<a name="L102" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L102">102</a>         calls to things like tasklet_schedule().)
<a name="L103" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L103">103</a> 2.      Convert all drivers that you must use from tasklets to workqueues.
<a name="L104" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L104">104</a> 3.      Force interrupts for drivers using tasklets onto other CPUs,
<a name="L105" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L105">105</a>         and also do I/O involving these drivers on other CPUs.
<a name="L106" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L106">106</a> SCHED_SOFTIRQ: Do all of the following:
<a name="L107" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L107">107</a> 1.      Avoid sending scheduler IPIs to the CPU to be de-jittered,
<a name="L108" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L108">108</a>         for example, ensure that at most one runnable kthread is present
<a name="L109" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L109">109</a>         on that CPU.  If a thread that expects to run on the de-jittered
<a name="L110" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L110">110</a>         CPU awakens, the scheduler will send an IPI that can result in
<a name="L111" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L111">111</a>         a subsequent SCHED_SOFTIRQ.
<a name="L112" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L112">112</a> 2.      Build with CONFIG_RCU_NOCB_CPU=y, CONFIG_RCU_NOCB_CPU_ALL=y,
<a name="L113" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L113">113</a>         CONFIG_NO_HZ_FULL=y, and, in addition, ensure that the CPU
<a name="L114" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L114">114</a>         to be de-jittered is marked as an adaptive-ticks CPU using the
<a name="L115" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L115">115</a>         "nohz_full=" boot parameter.  This reduces the number of
<a name="L116" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L116">116</a>         scheduler-clock interrupts that the de-jittered CPU receives,
<a name="L117" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L117">117</a>         minimizing its chances of being selected to do the load balancing
<a name="L118" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L118">118</a>         work that runs in SCHED_SOFTIRQ context.
<a name="L119" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L119">119</a> 3.      To the extent possible, keep the CPU out of the kernel when it
<a name="L120" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L120">120</a>         is non-idle, for example, by avoiding system calls and by
<a name="L121" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L121">121</a>         forcing both kernel threads and interrupts to execute elsewhere.
<a name="L122" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L122">122</a>         This further reduces the number of scheduler-clock interrupts
<a name="L123" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L123">123</a>         received by the de-jittered CPU.
<a name="L124" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L124">124</a> HRTIMER_SOFTIRQ:  Do all of the following:
<a name="L125" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L125">125</a> 1.      To the extent possible, keep the CPU out of the kernel when it
<a name="L126" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L126">126</a>         is non-idle.  For example, avoid system calls and force both
<a name="L127" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L127">127</a>         kernel threads and interrupts to execute elsewhere.
<a name="L128" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L128">128</a> 2.      Build with CONFIG_HOTPLUG_CPU=y.  Once boot completes, force the
<a name="L129" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L129">129</a>         CPU offline, then bring it back online.  This forces recurring
<a name="L130" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L130">130</a>         timers to migrate elsewhere.  If you are concerned with multiple
<a name="L131" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L131">131</a>         CPUs, force them all offline before bringing the first one
<a name="L132" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L132">132</a>         back online.  Once you have onlined the CPUs in question, do not
<a name="L133" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L133">133</a>         offline any other CPUs, because doing so could force the timer
<a name="L134" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L134">134</a>         back onto one of the CPUs in question.
<a name="L135" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L135">135</a> RCU_SOFTIRQ:  Do at least one of the following:
<a name="L136" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L136">136</a> 1.      Offload callbacks and keep the CPU in either dyntick-idle or
<a name="L137" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L137">137</a>         adaptive-ticks state by doing all of the following:
<a name="L138" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L138">138</a>         a.      Build with CONFIG_RCU_NOCB_CPU=y, CONFIG_RCU_NOCB_CPU_ALL=y,
<a name="L139" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L139">139</a>                 CONFIG_NO_HZ_FULL=y, and, in addition ensure that the CPU
<a name="L140" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L140">140</a>                 to be de-jittered is marked as an adaptive-ticks CPU using
<a name="L141" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L141">141</a>                 the "nohz_full=" boot parameter.  Bind the rcuo kthreads
<a name="L142" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L142">142</a>                 to housekeeping CPUs, which can tolerate OS jitter.
<a name="L143" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L143">143</a>         b.      To the extent possible, keep the CPU out of the kernel
<a name="L144" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L144">144</a>                 when it is non-idle, for example, by avoiding system
<a name="L145" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L145">145</a>                 calls and by forcing both kernel threads and interrupts
<a name="L146" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L146">146</a>                 to execute elsewhere.
<a name="L147" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L147">147</a> 2.      Enable RCU to do its processing remotely via dyntick-idle by
<a name="L148" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L148">148</a>         doing all of the following:
<a name="L149" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L149">149</a>         a.      Build with CONFIG_NO_HZ=y and CONFIG_RCU_FAST_NO_HZ=y.
<a name="L150" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L150">150</a>         b.      Ensure that the CPU goes idle frequently, allowing other
<a name="L151" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L151">151</a>                 CPUs to detect that it has passed through an RCU quiescent
<a name="L152" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L152">152</a>                 state.  If the kernel is built with CONFIG_NO_HZ_FULL=y,
<a name="L153" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L153">153</a>                 userspace execution also allows other CPUs to detect that
<a name="L154" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L154">154</a>                 the CPU in question has passed through a quiescent state.
<a name="L155" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L155">155</a>         c.      To the extent possible, keep the CPU out of the kernel
<a name="L156" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L156">156</a>                 when it is non-idle, for example, by avoiding system
<a name="L157" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L157">157</a>                 calls and by forcing both kernel threads and interrupts
<a name="L158" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L158">158</a>                 to execute elsewhere.
<a name="L159" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L159">159</a> 
<a name="L160" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L160">160</a> Name: kworker/%u:%d%s (cpu, id, priority)
<a name="L161" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L161">161</a> Purpose: Execute workqueue requests
<a name="L162" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L162">162</a> To reduce its OS jitter, do any of the following:
<a name="L163" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L163">163</a> 1.      Run your workload at a real-time priority, which will allow
<a name="L164" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L164">164</a>         preempting the kworker daemons.
<a name="L165" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L165">165</a> 2.      A given workqueue can be made visible in the sysfs filesystem
<a name="L166" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L166">166</a>         by passing the WQ_SYSFS to that workqueue's alloc_workqueue().
<a name="L167" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L167">167</a>         Such a workqueue can be confined to a given subset of the
<a name="L168" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L168">168</a>         CPUs using the /sys/devices/virtual/workqueue/*/cpumask sysfs
<a name="L169" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L169">169</a>         files.  The set of WQ_SYSFS workqueues can be displayed using
<a name="L170" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L170">170</a>         "ls sys/devices/virtual/workqueue".  That said, the workqueues
<a name="L171" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L171">171</a>         maintainer would like to caution people against indiscriminately
<a name="L172" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L172">172</a>         sprinkling WQ_SYSFS across all the workqueues.  The reason for
<a name="L173" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L173">173</a>         caution is that it is easy to add WQ_SYSFS, but because sysfs is
<a name="L174" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L174">174</a>         part of the formal user/kernel API, it can be nearly impossible
<a name="L175" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L175">175</a>         to remove it, even if its addition was a mistake.
<a name="L176" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L176">176</a> 3.      Do any of the following needed to avoid jitter that your
<a name="L177" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L177">177</a>         application cannot tolerate:
<a name="L178" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L178">178</a>         a.      Build your kernel with CONFIG_SLUB=y rather than
<a name="L179" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L179">179</a>                 CONFIG_SLAB=y, thus avoiding the slab allocator's periodic
<a name="L180" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L180">180</a>                 use of each CPU's workqueues to run its cache_reap()
<a name="L181" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L181">181</a>                 function.
<a name="L182" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L182">182</a>         b.      Avoid using oprofile, thus avoiding OS jitter from
<a name="L183" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L183">183</a>                 wq_sync_buffer().
<a name="L184" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L184">184</a>         c.      Limit your CPU frequency so that a CPU-frequency
<a name="L185" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L185">185</a>                 governor is not required, possibly enlisting the aid of
<a name="L186" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L186">186</a>                 special heatsinks or other cooling technologies.  If done
<a name="L187" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L187">187</a>                 correctly, and if you CPU architecture permits, you should
<a name="L188" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L188">188</a>                 be able to build your kernel with CONFIG_CPU_FREQ=n to
<a name="L189" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L189">189</a>                 avoid the CPU-frequency governor periodically running
<a name="L190" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L190">190</a>                 on each CPU, including cs_dbs_timer() and od_dbs_timer().
<a name="L191" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L191">191</a>                 WARNING:  Please check your CPU specifications to
<a name="L192" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L192">192</a>                 make sure that this is safe on your particular system.
<a name="L193" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L193">193</a>         d.      It is not possible to entirely get rid of OS jitter
<a name="L194" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L194">194</a>                 from vmstat_update() on CONFIG_SMP=y systems, but you
<a name="L195" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L195">195</a>                 can decrease its frequency by writing a large value
<a name="L196" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L196">196</a>                 to /proc/sys/vm/stat_interval.  The default value is
<a name="L197" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L197">197</a>                 HZ, for an interval of one second.  Of course, larger
<a name="L198" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L198">198</a>                 values will make your virtual-memory statistics update
<a name="L199" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L199">199</a>                 more slowly.  Of course, you can also run your workload
<a name="L200" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L200">200</a>                 at a real-time priority, thus preempting vmstat_update(),
<a name="L201" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L201">201</a>                 but if your workload is CPU-bound, this is a bad idea.
<a name="L202" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L202">202</a>                 However, there is an RFC patch from Christoph Lameter
<a name="L203" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L203">203</a>                 (based on an earlier one from Gilad Ben-Yossef) that
<a name="L204" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L204">204</a>                 reduces or even eliminates vmstat overhead for some
<a name="L205" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L205">205</a>                 workloads at https://lkml.org/lkml/2013/9/4/379.
<a name="L206" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L206">206</a>         e.      If running on high-end powerpc servers, build with
<a name="L207" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L207">207</a>                 CONFIG_PPC_RTAS_DAEMON=n.  This prevents the RTAS
<a name="L208" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L208">208</a>                 daemon from running on each CPU every second or so.
<a name="L209" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L209">209</a>                 (This will require editing Kconfig files and will defeat
<a name="L210" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L210">210</a>                 this platform's RAS functionality.)  This avoids jitter
<a name="L211" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L211">211</a>                 due to the rtas_event_scan() function.
<a name="L212" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L212">212</a>                 WARNING:  Please check your CPU specifications to
<a name="L213" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L213">213</a>                 make sure that this is safe on your particular system.
<a name="L214" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L214">214</a>         f.      If running on Cell Processor, build your kernel with
<a name="L215" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L215">215</a>                 CBE_CPUFREQ_SPU_GOVERNOR=n to avoid OS jitter from
<a name="L216" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L216">216</a>                 spu_gov_work().
<a name="L217" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L217">217</a>                 WARNING:  Please check your CPU specifications to
<a name="L218" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L218">218</a>                 make sure that this is safe on your particular system.
<a name="L219" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L219">219</a>         g.      If running on PowerMAC, build your kernel with
<a name="L220" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L220">220</a>                 CONFIG_PMAC_RACKMETER=n to disable the CPU-meter,
<a name="L221" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L221">221</a>                 avoiding OS jitter from rackmeter_do_timer().
<a name="L222" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L222">222</a> 
<a name="L223" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L223">223</a> Name: rcuc/%u
<a name="L224" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L224">224</a> Purpose: Execute RCU callbacks in CONFIG_RCU_BOOST=y kernels.
<a name="L225" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L225">225</a> To reduce its OS jitter, do at least one of the following:
<a name="L226" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L226">226</a> 1.      Build the kernel with CONFIG_PREEMPT=n.  This prevents these
<a name="L227" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L227">227</a>         kthreads from being created in the first place, and also obviates
<a name="L228" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L228">228</a>         the need for RCU priority boosting.  This approach is feasible
<a name="L229" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L229">229</a>         for workloads that do not require high degrees of responsiveness.
<a name="L230" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L230">230</a> 2.      Build the kernel with CONFIG_RCU_BOOST=n.  This prevents these
<a name="L231" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L231">231</a>         kthreads from being created in the first place.  This approach
<a name="L232" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L232">232</a>         is feasible only if your workload never requires RCU priority
<a name="L233" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L233">233</a>         boosting, for example, if you ensure frequent idle time on all
<a name="L234" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L234">234</a>         CPUs that might execute within the kernel.
<a name="L235" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L235">235</a> 3.      Build with CONFIG_RCU_NOCB_CPU=y and CONFIG_RCU_NOCB_CPU_ALL=y,
<a name="L236" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L236">236</a>         which offloads all RCU callbacks to kthreads that can be moved
<a name="L237" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L237">237</a>         off of CPUs susceptible to OS jitter.  This approach prevents the
<a name="L238" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L238">238</a>         rcuc/%u kthreads from having any work to do, so that they are
<a name="L239" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L239">239</a>         never awakened.
<a name="L240" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L240">240</a> 4.      Ensure that the CPU never enters the kernel, and, in particular,
<a name="L241" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L241">241</a>         avoid initiating any CPU hotplug operations on this CPU.  This is
<a name="L242" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L242">242</a>         another way of preventing any callbacks from being queued on the
<a name="L243" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L243">243</a>         CPU, again preventing the rcuc/%u kthreads from having any work
<a name="L244" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L244">244</a>         to do.
<a name="L245" href="http://lxr.free-electrons.com/source/Documentation/kernel-per-CPU-kthreads.txt#L245">245</a>
